# Claude Code 项目四件套实操配置指南

适用场景：

* 从零创建新的本地 Python AI 项目
* 迭代已有的本地 Python AI 项目

核心目标：

> 让 Claude Code 清楚知道项目要做什么、该怎么安全地做、当前做到哪、每次任务如何执行。

推荐四件套：

```text
/docs/project_brief.md      # 项目需求基线：定义做什么
CLAUDE.md                   # Claude Code 行为规范：定义怎么做
/docs/project_status.md     # 项目状态记录：定义做到哪
固定任务模板                 # 每次任务的执行格式
```

---

# 一、四件套的关系

## 1. `/docs/project_brief.md`

解决问题：

```text
这个项目要做什么？
```

它是需求基线文档，告诉 Claude Code：

* 项目目标
* 核心功能
* MVP 范围
* 不做什么
* 技术约束
* 成本与安全约束
* 验收标准

---

## 2. `CLAUDE.md`

解决问题：

```text
Claude Code 在这个项目里应该怎么工作？
```

它是 Claude Code 的项目级规则文件，告诉 Claude：

* 必读哪些文件
* 哪些操作可自动执行
* 哪些操作必须确认
* 如何节省上下文和 token
* 如何运行测试
* 如何维护项目文档
* 如何避免危险操作

---

## 3. `/docs/project_status.md`

解决问题：

```text
项目现在做到哪了？
```

它是项目长期状态记录，告诉 Claude Code：

* 已完成什么
* 正在做什么
* 还有哪些遗留问题
* 下一步优先级是什么
* 最近任务改了哪些文件
* 测试结果如何

---

## 4. 固定任务模板

解决问题：

```text
这一次具体要 Claude Code 做什么？
```

它用于每次下达任务，明确：

* 本次目标
* 修改范围
* 禁止操作
* 测试要求
* 文档更新要求
* 最终输出格式

---

# 二、推荐执行顺序

## 场景 A：从零创建新项目

推荐顺序：

```text
1. 创建 /docs/project_brief.md
2. 创建 CLAUDE.md
3. 创建 /docs/project_status.md
4. 使用固定任务模板开始第一个小任务
```

为什么：

* 新项目还没有代码，先写需求基线，避免 Claude 自由发挥
* 再写 `CLAUDE.md`，约束 Claude 的行为、安全边界和输出方式
* 再写 `project_status.md`，建立项目长期记录
* 最后用任务模板小步开发

---

## 场景 B：迭代已有项目

推荐顺序：

```text
1. 创建 CLAUDE.md
2. 让 Claude Code 只读分析已有项目
3. 生成或补全 /docs/project_brief.md
4. 生成或补全 /docs/project_status.md
5. 使用固定任务模板继续迭代
```

为什么：

* 已有项目最怕误改，所以先创建 `CLAUDE.md` 限制行为
* 再让 Claude 只读分析，不要直接修改业务代码
* 然后反向整理需求基线和项目状态
* 最后进入正常迭代流程

---

# 三、第一件事：创建 `/docs/project_brief.md`

## 为什么要做

`project_brief.md` 是项目需求基线。

没有它，Claude Code 容易：

* 只根据局部代码猜项目目标
* 对功能做过度设计
* 引入不必要依赖
* 修 bug 时破坏原始需求
* 多轮任务之间标准不一致

有了它，Claude 每次任务都能先对齐项目目标。

---

## 文件位置

```text
/docs/project_brief.md
```

建议使用这个名字，而不是 `requirements.md`。

原因：

* `requirements.md` 容易和 `requirements.txt` 混淆
* `prd.md` 对个人开发者偏重
* `project_brief.md` 更轻量，适合作为 Claude Code 的需求基线

---

## 从零创建新项目时怎么做

你可以先手写一个简版，只要写清楚：

```text
项目目标
MVP 功能
暂不做什么
技术约束
验收标准
```

然后让 Claude Code 基于它初始化项目。

推荐指令：

```text
请阅读 /docs/project_brief.md。

任务：
基于 project_brief.md，帮我检查需求是否足够支持创建最小项目骨架。

要求：
1. 不要创建代码
2. 不要做过度设计
3. 只指出缺失信息、风险点和建议的 MVP 开发顺序
4. 不确定的内容标记为“待确认”
```

---

## 迭代已有项目时怎么做

已有项目可以让 Claude Code 反向生成。

推荐指令：

```text
请基于当前已有项目，生成 /docs/project_brief.md。

要求：
1. 只读分析，不要修改业务代码
2. 优先阅读 README、pyproject.toml、requirements.txt、src/、tests/ 中必要文件
3. 不做无意义全仓扫描
4. 根据已有代码反向总结项目目标、核心功能、技术约束和当前优先级
5. 不要编造不存在的功能
6. 不确定的信息标记为“待确认”
7. 创建或更新 /docs/project_brief.md
```

# 四、第二件事：创建 `CLAUDE.md`

## 为什么要做

`CLAUDE.md` 是 Claude Code 的项目规则文件。

它用于告诉 Claude：

* 每次任务前先看哪些文件
* 哪些操作可以自动执行
* 哪些操作必须停下来确认
* 如何减少不必要输出
* 如何维护项目状态
* 如何处理测试和文档

一句话：

> `project_brief.md` 管目标，`CLAUDE.md` 管行为。

---

## 文件位置

```text
CLAUDE.md
```

放在项目根目录。

---

## 创建方式

可以手动创建，也可以在 Claude Code 里使用：

```text
/init
```

如果使用 `/init` 自动生成，建议再手动改成下面这版。

---

## 推荐模板

```markdown
# CLAUDE.md

本项目是一个本地 Python AI 项目。Claude Code 在本仓库中工作时，必须遵守本文件规则。

---

## 1. 必读文件

执行任务前，根据任务需要优先阅读：

- /docs/project_brief.md
- /docs/project_status.md
- 与本次任务直接相关的源码和测试

不要无意义全仓扫描。

---

## 2. 工作原则

- 只完成用户指定任务，不做无关扩展
- 优先阅读必要文件
- 修改前先理解相关上下文
- 修改后运行相关测试
- 输出保持简洁
- 不输出长篇推理
- 不重复解释用户已经确认过的内容
- Python 项目统一使用 `uv`，作为 Python 包管理器和环境管理工具

---

## 3. 安全底线

未经用户确认，禁止：

- 删除核心代码文件
- 覆盖配置文件
- 修改生产环境配置
- 数据库结构变更
- 批量删除文件或数据
- 执行高风险系统命令
- 大规模重构
- 修改认证、权限、密钥、支付相关逻辑
- 把敏感信息写入日志、文档或测试快照

---

## 4. 可自动执行的操作

以下操作可直接执行：

- 新增普通业务代码
- 修改非核心逻辑
- 新增或修改测试
- 更新普通文档
- 代码格式化
- 小范围重构
- 修复明确 bug
- 增加类型标注
- 改善错误提示或日志

---

## 5. 需要先说明但可继续执行的操作

以下操作需要先说明影响范围和修改方案；如不触及安全底线，可继续执行：

- 修改多个模块
- 调整核心流程
- 引入新依赖
- 修改项目结构
- 改动数据库访问逻辑
- 修改 AI pipeline 关键步骤
- 修改模型调用逻辑
- 修改 prompt 组织方式
- 修改缓存、队列、并发逻辑

---

## 6. 必须人工确认的操作

遇到以下情况必须停止并等待确认：

- 删除核心代码文件
- 覆盖配置文件
- 修改生产环境配置
- 数据库结构变更
- 批量删除文件或数据
- 执行高风险系统命令
- 大规模重构
- 不可逆操作
- 可能产生明显费用的外部 API 调用
- 可能上传本地文件或敏感数据的操作

---

## 7. Claude Code 使用规则

优先使用以下 Claude Code 工作方式：

- 大改前先使用 plan mode
- 长任务前先拆分步骤
- 修改完成后查看 diff
- 任务变长时及时 compact 上下文
- 任务结束前进行自检
- 必要时使用 code review 检查改动

避免：

- 在未确认时使用危险权限模式
- 在未隔离环境中跳过权限检查
- 使用 hooks 执行高风险命令
- 让 Claude 自行决定删除或批量移动重要文件

---

## 8. Token 与上下文节省规则

- 优先阅读必要文件，不做无意义全仓扫描
- 优先基于当前任务做增量分析
- 不重复解释已确认内容
- 不粘贴完整代码，除非用户要求
- 优先引用文件路径、函数名和修改点
- 测试结果只输出关键结论和失败原因
- 长期上下文写入 /docs/project_status.md
- 需求基线写入 /docs/project_brief.md

---

## 9. 文档维护规则

每次完成独立功能、修复、优化或重构后，必须增量更新：

/docs/project_status.md

要求：

- 不全文重写
- 在文件末尾追加本次任务记录
- 包含目标、完成内容、修改文件、实现思路、遗留问题、下一步、关键指令、测试结果、更新时间

当需求、范围、优先级发生变化时，应更新：

/docs/project_brief.md

要求：

- 只做局部更新
- 不删除重要约束
- 不编造未经确认的新需求

---

## 10. 测试规则

优先运行与本次修改直接相关的测试。

如果无法运行测试，必须说明：

- 为什么无法运行
- 哪些部分未验证
- 建议用户如何验证

---

## 11. 输出格式

任务结束时只输出：

- 修改文件列表
- 核心变更摘要
- 测试结果
- 遗留问题
- 下一步建议
```

---

## 建议执行一次检查

在 Claude Code 中运行：

```text
/memory
```

确认 `CLAUDE.md` 已被加载。

# 六、第四件事：每次任务使用固定模板

## 为什么要做

固定任务模板可以减少沟通成本和误操作。

没有模板时，Claude Code 容易：

* 不知道允许改哪些文件
* 做无关优化
* 每一步都请求确认
* 忘记跑测试
* 忘记更新文档
* 输出过长，浪费上下文

固定模板的作用：

* 明确本次目标
* 明确修改边界
* 明确安全规则
* 明确测试要求
* 明确文档更新要求
* 明确输出格式

---

# 6.1 通用任务模板

适合大多数任务。

```text
请基于 CLAUDE.md、/docs/project_brief.md 和 /docs/project_status.md 执行本次任务。

任务目标：
【填写具体任务】

背景说明：
【说明当前问题、期望结果、相关模块；没有可写“无”】

允许修改：
- 【文件或目录 1】
- 【文件或目录 2】

禁止操作：
- 不要删除核心代码文件
- 不要覆盖配置文件
- 不要修改生产环境配置
- 不要执行高风险系统命令
- 不要做无关重构
- 不要上传本地敏感文件
- 不要记录 API key 或敏感信息

执行要求：
1. 只阅读完成任务所需的必要文件
2. 如改动较大，先进入计划模式并输出简短方案
3. 低风险修改可直接执行
4. 中等风险修改先说明影响范围；不触及安全底线可继续执行
5. 高风险操作必须等待确认
6. 修改后运行相关测试
7. 完成后查看 diff 并自检
8. 完成后增量更新 /docs/project_status.md
9. 如果需求范围发生变化，再局部更新 /docs/project_brief.md

Token 与上下文要求：
1. 不做无意义全仓扫描
2. 不输出长篇推理
3. 不粘贴完整代码，除非我要求
4. 优先说明文件路径、函数名和修改点
5. 测试结果只输出关键结论和失败原因

最终输出：
- 修改文件列表
- 核心变更摘要
- 测试结果
- 遗留问题
- 下一步建议
```

# 八、权限与安全建议

## 推荐权限策略

个人本地开发建议采用：

```text
低风险自动执行
中风险先说明后执行
高风险必须确认
```

---

## 可自动执行

* 读取项目文件
* 修改普通业务代码
* 新增测试
* 修改测试
* 更新普通文档
* 运行单元测试
* 运行格式化命令
* 小范围重构

---

## 需要先说明但可继续执行

* 修改多个模块
* 引入新依赖
* 修改项目结构
* 调整核心流程
* 修改模型调用逻辑
* 修改 RAG pipeline
* 改动数据库访问逻辑

---

## 必须人工确认

* 删除核心代码文件
* 批量删除文件
* 覆盖配置文件
* 修改生产环境配置
* 数据库结构变更
* 执行高风险系统命令
* 上传本地文件
* 执行可能产生明显费用的 API 调用
* 使用危险权限模式
* 添加 hooks 执行 shell 命令

---

## 关于跳过权限检查

不建议在普通本地项目中跳过权限检查。

只有在以下隔离环境中才考虑：

* 临时容器
* 虚拟机
* dev container
* 无敏感文件
* 无生产密钥
* 无重要本地数据

即使在隔离环境中，也建议保留：

* 不删除核心文件
* 不批量删除
* 不修改生产配置
* 不上传敏感数据

---

## 关于 hooks

hooks 可以自动执行固定命令，但风险较高。

适合：

* 自动运行格式化
* 自动运行 lint
* 自动记录日志
* 自动阻止某些危险操作

不适合：

* 删除文件
* 修改配置
* 自动提交代码
* 自动推送代码
* 访问密钥或私有数据

建议原则：

```text
能不用 hooks 就先不用。
需要 hooks 时，只做简单、可审计、低风险命令。
```

# 十一、最小配置总结

最小四件套：

```text
1. /docs/project_brief.md
2. CLAUDE.md
3. /docs/project_status.md
4. 固定任务模板
```

四件套关系：

```text
project_brief.md  = 做什么
CLAUDE.md         = 怎么做、哪些不能做
project_status.md = 做到哪、下一步是什么
任务模板           = 这一次具体做什么
```

日常使用方式：

```text
1. 让 Claude 先读 CLAUDE.md
2. 再读 project_brief.md 确认目标
3. 再读 project_status.md 确认当前状态
4. 用固定模板下达单个小任务
5. 任务完成后查看 diff
6. 运行测试
7. 更新 project_status.md
8. 需求变了才更新 project_brief.md
```

一句话原则：

> 需求写进 `project_brief.md`，规则写进 `CLAUDE.md`，进度写进 `project_status.md`，每次任务用固定模板小步推进。
